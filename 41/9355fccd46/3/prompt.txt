You are taking over Nimbus “Minimal Report UI V1” implementation.

Context:
- Product direction: keep GitHub/CI-native workflow, but add a lightweight visual report viewer.
- We do NOT want a full dashboard yet.
- We want an easy shareable/readable report page with strong copy-to-agent UX.
- Spec reference: llm-docs/minimal-report-ui-v1.md


Your mission:
Build a minimal report viewer experience end-to-end on a throwaway branch with sensible defaults.


Non-negotiable goals:
1) Render a review by reviewId from existing worker API.
2) Make it easy to copy report content into coding agents.
3) Keep setup simple and avoid overbuilding.


Implementation defaults (already decided):
- URL route: /reports/:reviewId
- Data source: existing API GET /api/reviews/:id
- V1 has no auth/SSO complexity (public-by-reviewId is acceptable for now)
- Show:
  - summary header (recommendation, risk, findings count, status, timestamps)
  - findings cards
  - evidence list
  - rendered markdown summary
  - raw JSON (collapsible)
- Copy actions:
  - copy full markdown
  - copy full JSON
  - copy per-finding text
  - copy per-finding “fix prompt”
- Use this fix prompt template per finding:


  You are helping fix a Nimbus code review finding.


  Finding title: <title>
  Severity: <severity>
  Confidence: <confidence>
  Description:
  <description>


  Conditions:
  <conditions or "none provided”>


  Locations:
  <path:line list>


  Suggested fix:
  <suggestedFix or "not provided”>


  Evidence refs:
  <evidence refs list>


  Please:
  1) Propose a minimal safe code change.
  2) Explain why it resolves the issue.
  3) List any tests to run.
  4) Return a patch-style diff when possible.


- UX defaults:
  - neutral clean style, readability-first
  - toast feedback on copy (“Copied”)
  - robust loading/empty/error states
- Technical defaults:
  - use existing web stack if present in repo; if absent, choose lightweight React+Vite
  - sanitize markdown rendering
  - static-host-friendly architecture


Questions you might ask (already answered):
- Should we include auth? → No, not in V1.
- Should we build list/history pages? → No, single-report page only.
- Should we support downloads? → Yes, markdown/json download buttons.
- Should we include feedback buttons? → No, not in V1.
- Should we optimize for branding? → No, optimize for clarity and utility.
- Should we support queued/failed reviews? → Yes, first-class UI states.


Execution style:
- Work in vertical slices and run tests after each meaningful slice.
- Reuse existing project patterns and keep changes surgical.
- If you find no existing frontend target, create a small self-contained one with clear run instructions.
- Add/update tests where practical (component/unit at minimum).
- Include local run instructions and a quick smoke checklist.


Deliverables:
1) Implemented report UI route/page.
2) Copy buttons (full + finding + fix prompt) working.
3) Download markdown/json working.
4) Tests green.
5) Final report with:
   - files changed
   - commands run
   - test results
   - any follow-ups for post-V1.

Do not forget to make a new branch for this.
