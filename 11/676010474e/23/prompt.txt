# Nimbus Checkpoint Deployment Plan

## Goal

Enable Nimbus to deploy websites directly from **Entire committed checkpoints** with reliable, reproducible build and deploy behavior.

This path should feel like:

1. Pick checkpoint (or commit)
2. Build exactly that source snapshot
3. Deploy
4. Keep a durable audit trail (checkpoint, commit, logs, artifacts, URL)

---

## Product Outcomes

- Deterministic deployments from immutable source snapshots
- Strong provenance: deployment is linked to `Entire-Checkpoint` and commit SHA
- Fast debug cycle with summarized logs and raw logs
- Clear separation between source resolution and runtime/build/deploy orchestration

---

## MVP Scope

### In Scope

- Deploy from **committed** Entire checkpoints only
- Accept either:
  - checkpoint ID (12-char hex)
  - commit-ish (`<sha>`, branch ref, tag)
- Resolve to commit SHA, create source bundle from Git tree, upload to worker
- Build/test/lint/deploy/smoke pipeline in sandbox
- Persist deployment metadata, artifacts, and logs
- CLI support for checkpoint deployment and status watch

### Out of Scope (Phase 2)

- Temporary shadow checkpoint deployments
- In-browser checkpoint selection UI
- Multi-checkpoint batch deployments
- Differential deployment optimization between checkpoints

---

## Why Committed Checkpoints First

Committed checkpoints are ideal deployment inputs because they map to immutable Git commits and are globally reproducible. They avoid local-only state and simplify source resolution, artifact integrity, and rollback.

---

## End-to-End Flow

1. User runs checkpoint deploy command in CLI
2. CLI resolves checkpoint -> commit SHA and collects checkpoint metadata
3. CLI archives source from that commit (`git archive`) into a tarball
4. CLI calls worker create-job endpoint with checkpoint metadata + source bundle
5. Worker stores bundle in R2 and enqueues job execution
6. Sandbox unpacks source, runs install/build/validation
7. On pass, worker deploys to Pages and runs runtime smoke check
8. Worker emits SSE events and stores logs/artifacts/metrics
9. CLI renders progress and final output URLs + provenance details

---

## Source Resolution Model

### Accepted Inputs

- `checkpoint:<id>` (explicit)
- `<id>` if it matches checkpoint ID pattern
- `commit:<sha-or-ref>` (explicit)
- raw commit-ish argument

### Resolution Rules

1. If input is checkpoint ID:
   - Find commit containing trailer `Entire-Checkpoint: <id>`
   - If multiple matches, choose newest reachable commit on current branch by default
2. If input is commit-ish:
   - Resolve directly via `git rev-parse`
   - Parse commit message trailers; attach checkpoint metadata if present
3. If no checkpoint trailer exists on commit:
   - Deployment can still proceed (commit deploy)
   - Mark `checkpointId = null`

### Provenance Captured

- `checkpointId` (nullable)
- `commitSha` (required)
- `branchRef` (best effort)
- `entireSessionId` (if discoverable from metadata)
- `entireAttribution` trailer (if present)

---

## API Contract (Checkpoint Source)

### Create Job

`POST /api/jobs`

Use `multipart/form-data`:

- `metadata` (JSON string)
- `bundle` (tar.gz source archive)

Metadata shape:

```json
{
  "source": {
    "type": "checkpoint",
    "checkpointId": "8a513f56ed70",
    "commitSha": "abc123...",
    "ref": "v2-agentic"
  },
  "build": {
    "runTestsIfPresent": true,
    "runLintIfPresent": true
  }
}
```

Response `202`:

```json
{
  "jobId": "job_xxxxxxxx",
  "status": "queued",
  "phase": "queued",
  "eventsUrl": "/api/jobs/job_xxxxxxxx/events",
  "jobUrl": "/api/jobs/job_xxxxxxxx"
}
```

### Stream Events

`GET /api/jobs/:id/events?from=<seq>`

- at-least-once delivery
- replay from `Last-Event-ID` and/or `from`

### Job Details

`GET /api/jobs/:id`

- include source provenance fields (`sourceType`, `checkpointId`, `commitSha`)

### Logs

`GET /api/jobs/:id/logs?format=summary|raw`

- successful deploy from checkpoint

Tests:

- deploy gating
- smoke fail semantics
- final summary payload includes provenance fields

### Slice 5: Retry/Cancel + Operational Hardening

Build:

- retry/cancel action path compatibility for checkpoint jobs
- reconnect-safe event streaming in CLI watch flow
- log retrieval and summaries from API

- cancel in-flight build
- retry from failed build using same source bundle
- reconnect watch without duplicate rendering

Tests:

- cancellation race behavior
- retry state transition integrity
- SSE replay dedupe logic

## Security and Safety Controls

- Enforce max source bundle size
- Reject path traversal during archive extraction
- Keep safer install mode as default, then controlled fallback
- Persist full raw logs and deterministic summary logs separately
- Restrict accepted source type to `checkpoint` for this phase

- install/build/test/lint/deploy/smoke durations
- total job duration
- retry count
- artifact byte totals
- success/failure rates by phase
