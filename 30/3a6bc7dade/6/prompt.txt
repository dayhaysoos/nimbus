They must be introduced distinctly from changed files. The prompt should make clear that related files are historically co-changed with the modified code (surfaced from Entire checkpoint session history), not directly modified in this commit. The model needs this distinction to correctly weight related file context — without it, related files get treated the same as changed files and the co-change signal is diluted.

Context for why I'm asking: We're considering introducing OpenRouter as the model provider for review analysis. OpenRouter is OpenAI API-compatible, accepts a single API key, and gives users access to Claude, GPT, Gemini, Mistral and others without needing provider-specific keys. The integration would replace or sit alongside the Cloudflare AI SDK for the model call layer specifically.


The intended design is a thin `ReviewModelClient` interface:


interface ReviewModelClient {
  complete(prompt: string): Promise<string>
}


With provider implementations injected via config — starting with an `OpenRouterModelClient` that calls `https://openrouter.ai/api/v1/chat/completions` using a standard fetch with the user's `OPENROUTER_API_KEY` env var. This abstraction means the review runner never knows which provider it's talking to, and adding providers later is just a new interface implementation.


The goal is reducing adoption friction — one OpenRouter key works with any supported model and is simpler to set up than Cloudflare AI bindings or provider-specific keys. This also naturally aligns with Phase 4 integration surface work.
