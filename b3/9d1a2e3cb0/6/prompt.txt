Let’s go ahead start specing out how we want to build out the context retrieval part of @llm-docs/review-quality-phases-context-prompt-integration.md .

We're specing out Phase 1 of Nimbus review quality improvements: Context Retrieval.

The goal is that review quality should not depend on diff hunks alone. We're introducing a `ReviewContext` type that gets assembled at `review create` time and passed to the review agent.


**Hard constraint**: Nimbus only supports Entire checkpoints. There is no fallback to static analysis or generic repo scanning. The co-change signal from Entire session history is the core retrieval mechanism, not an enhancement.


**What ReviewContext should contain:**


- Diff hunks (baseline, always present)
- Full text of changed files
- Related files derived from co-change frequency in prior Entire checkpoint sessions — given the set of changed files in this review, scan recent checkpoint sessions on the `entire/checkpoints/v1` branch, find sessions that touched any of those files, collect all other files touched in those sessions, rank by co-change frequency, include top N
- Convention/guidance files discovered by walking from each changed file's directory up to repo root, matching a known pattern list (AGENTS.md, CODE_REVIEW.md, .eslintrc*, biome.json, pyproject.toml, etc.)
- Checkpoint context from the Entire session linked to this commit: sessionIntent (from prompt.txt or context.md), agentType, and attribution trailer


**Token budget**: No budget by default. Include everything and let the model's context window be the natural hard ceiling. If a user optionally sets a budget via config and the assembled ReviewContext exceeds it, hard fail with a clear message telling them to increase the budget. Do not silently degrade or drop context. Token usage is tracked in the provenance manifest as a feedback signal — large token usage indicates a large commit, not a tool configuration problem.


**Cloudflare platform specifics:**


- **ReviewContext assembly belongs in a Workflow step**, not inline in a Worker request handler. Assembly involves multiple async operations (GitHub API calls to fetch checkpoint branch data, file fetches, convention file walks) that can exceed Worker CPU limits. The existing Workflow/queue architecture for workspace and deploy processing should be extended to include a `review_context_assembly` step before the review agent pass.
- **Co-change lookup cache goes in D1**. D1 is already used as the durable audit log. Add a table to cache co-change frequency data keyed by file path so repeated lookups against the `entire/checkpoints/v1` branch don't re-fetch GitHub on every review. Cache entries should include a `last_updated` timestamp so staleness can be detected.
- **Large ReviewContext payloads (full file contents) should not be passed inline through the queue**. Store the assembled ReviewContext in Durable Object storage or R2, pass only a reference ID through the Workflow. The review agent step then fetches it by ID. This avoids queue message size limits and keeps the Workflow message lightweight.
- **The `entire/checkpoints/v1` branch queries are outbound GitHub API calls** from the Worker. Handle rate limiting explicitly — cache aggressively in D1, and surface GitHub API errors clearly in the provenance manifest rather than silently producing incomplete context.
- **Provenance manifest events should stream via the existing SSE infrastructure** already used for `review events`. As ReviewContext assembly progresses (diff fetched, co-change lookup complete, convention files found), emit incremental provenance events so the report UI can show assembly status live before the agent pass begins.


**Implementation approach for co-change lookup**: This is a lookup, not a graph. No graph library needed. For a given review, query the `entire/checkpoints/v1` branch session data, find sessions that touched any of the current changed files, collect co-touched files, rank by frequency, take top N. Cache results in D1.


**Spec should cover:**
1. The `ReviewContext` type contract (TypeScript)
2. The Workflow step design for assembly — where it fits in the existing workspace → deploy → review lifecycle
3. The co-change lookup implementation against `entire/checkpoints/v1` and D1 cache schema
4. Convention file discovery strategy
5. R2 or DO storage contract for ReviewContext payload — reference ID passing through Workflow
6. Optional token budget config, enforcement, and hard fail behavior
7. The provenance manifest shape, including SSE event emissions during assembly


Keep retrieval read-only and transparent. ReviewContext is an internal type, not user-facing, but its provenance section surfaces in the report.
